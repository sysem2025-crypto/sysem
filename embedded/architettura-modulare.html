    Architettura Modulare per PIC16Cxx

    Organizzare il codice Assembly su pagine ROM con moduli #include, header di dipendenza e controllo dimensionale

    Problema

    Nei microcontrollori PIC a 8 bit con architettura Harvard, la memoria programma è organizzata in pagine da 512 word. Saltare da una pagina all’altra richiede la gestione esplicita del registro PCLATH. Senza una struttura modulare chiara, il codice diventa rapidamente ingestibile: bisogna tracciare manualmente quanta ROM occupa ogni modulo, evitare overlap tra pagine, e garantire che le chiamate tra pagine funzionino correttamente.

    Soluzione: Moduli #include con tracciamento dimensionale

    I progetti più articolati dell’archivio ICA adottano una tecnica elegante: ogni modulo funzionale è un file .h incluso via #include nel main.asm, separato per pagina ROM. Alla fine di ogni pagina, delle costanti calcolano automaticamente lo spazio residuo:

    Struttura del main.asm (Split Elettroplastica, I200052)

    ;----------------------------------------------------------------------
; Il main.asm definisce l'ordine di inclusione e il mapping a pagina ROM
;----------------------------------------------------------------------
    ORG      4

I_01
    #modulo     "#Inter.h"    ; gestione interrupt
I_02
    include     "table.h"     ; tabelle operative
I_03
    include     "#Init.h"     ; general startup
I_04
    include     "#OS.h"       ; sistema operativo (scheduler)
I_05
    include     "#Triac.h"    ; innesco triac ON/OFF
I_06
    include     "#Timer.h"    ; timer interni
I_07
    include     "#Key.h"      ; lettura tasto
I_08
    include     "#Tacho.h"    ; lettura tachimetrica
I_09
    include     "#Ntc.h"      ; sonde di temperatura
I_10
    include     "#Buzzer.h"   ; buzzers
I_11
    include     "#Dcm.h"      ; decodifica comando
I_12
    include     "#EEP.h"      ; EEPROM esterna
I_13
    include     "#Flap.h"     ; controllo deflettore
I_14
    include     "#Diag.h"     ; diagnostica seriale
I_15
    ORG         0x800         ; cambio pagina ROM

    include     "table1.h"    ; tabelle operative pagina 1
I_16
    #modulo     "#M_State.h"  ; macchina a stati (1569 righe!)
I_17

; Controllo dimensionale automatico
General_Interrupt  SET  I_02 - I_01
Tabelle            SET  I_03 - I_02
Sistema_Operativo  SET  I_05 - I_04
Innesco_Triac      SET  I_06 - I_05
Sonode_Temperatura SET  I_10 - I_09
Macchina_a_Stati   SET  I_17 - I_16

ROM_PAGE_0  SET  0x800 - I_15    ; spazio residuo pagina 0
ROM_PAGE_1  SET  0xfff - I_17    ; spazio residuo pagina 1

    Ogni etichetta I_xx funge da marcatore. Le costanti calcolano automaticamente la dimensione di ogni modulo e lo spazio residuo per pagina. Se un modulo cresce troppo e sfora la pagina, l’assemblatore segnala un errore in fase di compilazione.

    La direttiva #modulo

    I progetti più recenti introducono una macro #modulo che funge da header decorativo con nome del modulo, e le direttive #if/#endif per escludere moduli non necessari a seconda della configurazione hardware:

    #modulo     "#M_State.h"
; Il nome del modulo viene stampato nel listato .lst
; rendendo immediata la navigazione del codice assemblato.

DECODE_MACRO    EQU  1
TLC_FERROLI     EQU  0    ; Telecomando Elettroplastica

    IF TLC_FERROLI == 1
        include  "#Tlc_Ferroli.h"
    ELSE
        include  "#Tlc_Elettroplastica.h"
    ENDIF

    Gestione pagine con GOTO_P e CALL_P

    Saltare tra pagine ROM richiede la gestione di PCLATH. I progetti usano macro come GOTO_P e CALL_P per astrarre questo dettaglio:

    ; Esempio da TASK_1.H (Antisbilanciamento I200026)
    BSF     STATUS,PA0
    BSF     STATUS,PA1
    GOTO_P  STATE_0       ; salto a pagina opportuna

; La macro GOTO_P si occupa di:
; 1. Caricare PCLATH con la pagina di destinazione
; 2. Eseguire il GOTO

; Moduli organizzati per pagina:
; Pagina 0: Init, NTC, Tacho
; Pagina 1: SM (state machine), Key, Sync, Triac, Sweep, Step
; Pagina 2: Cmd, Prepr, Postpr, Usr, DataOut, Test
; Pagina 3: Reg, Pla, Tlc

    L’organizzazione segue un criterio logico: inizializzazione e lettura sensori in pagina 0, logica di controllo e attuatori in pagina 1 e 2, comunicazione e diagnostica in pagina 3.

    Schema a blocchi

    main.asm
  |
  +-- Pagina 0 (0x000 - 0x1FF)
  |     #Inter.h    Gestione interrupt (salvataggio contesto, decoder)
  |     #Init.h     Inizializzazione periferiche e variabili
  |     #OS.h       Scheduler cooperativo round-robin
  |     #Triac.h    Innesco triac a tempo
  |     #Timer.h    Timer software interni
  |     #Key.h      Lettura e debounce tastiera
  |     #Tacho.h    Lettura tachimetrica (sensore Hall)
  |     #Ntc.h      Lettura temperatura NTC (metodo RC)
  |
  +-- Pagina 1 (0x200 - 0x3FF)
  |     #SM.h       Macchina a stati del prodotto
  |     #Key.h      Controllo pulsanti (copia pagina 0 se serve)
  |     #Sync.h     Sincronismo di rete
  |     #Triac.h    Innesco triac (continuazione)
  |     #Sweep.h    Controllo motore PP alto livello
  |     #Step.h     Controllo motore PP basso livello
  |
  +-- Pagina 2 (0x400 - 0x5FF)
  |     #Cmd.h      Decodifica comandi utente
  |     #Prepr.h    Preprocessing dati
  |     #Postpr.h   Postprocessing
  |     #Usr.h      Controllo compressore/utenze
  |     #DataOut.h  Trasmissione dati e autodiagnosi
  |     #Test.h     Diagnostica iniziale all'avvio
  |
  +-- Pagina 3 (0x600 - 0x7FF)
        #Reg.h      Registri timing
        #Pla.h      PLA (Programmable Logic Array)
        #Tlc.h      Controllo TLC (telecomando)

    Lezioni apprese

    • Separazione netta per pagina: ogni pagina ROM contiene moduli logicamente coerenti (sensori, controllo, comunicazione)
    • Controllo dimensionale automatico: le costanti derivate da etichette marcatore rilevano overlap in compilazione
    • Modularità fisica: ogni file .h = un modulo funzionale, con header commentato che dichiara dipendenze, risorse e allocazione
    • Macro di navigazione: GOTO_P e CALL_P rendono il cambio pagina trasparente
    • Direttiva #modulo: decorazione del listato per debug e manutenzione
