    Gestione Interrupt su PIC16Cxx

    Salvataggio contesto, interrupt on-change, decoder a cascata e sincronismo per triac, tacho e temperatura

    Problema

    Su un PIC16C65A, un singolo vettore interrupt all’indirizzo 0x004 serve tutte le periferiche: cambio stato su PORTB, overflow timer0, timer1, timer2. Ogni fonte deve essere identificata e servita in tempi strettissimi, senza perdere eventi. Inoltre il salvataggio del contesto (W, STATUS, PCLATH, FSR) deve essere completo ma velocissimo, e bisogna gestire l’annidamento implicito tra interrupt e task dello scheduler.

    Soluzione

    Il modulo #Inter.h implementa un decoder a cascata che identifica la fonte dell’interrupt e instrada il controllo al gestore appropriato, il tutto in meno di 10 cicli macchina. Il salvataggio contesto usa SWAPF per preservare STATUS senza alterarlo.

    Salvataggio contesto (prologo interrupt)

    ;----------------------------------------------------------------------
; Prologo interrupt: salva contesto in 4 istruzioni
;----------------------------------------------------------------------
Interrupt
PUSH
    movwf   Reg_w              ; salva W (nessun effetto su STATUS)
    swapf   STATUS,w           ; scambia nibble per salvare STATUS
    BANK    0
    movwf   Reg_Status         ; Reg_Status = STATUS preservato

    movfw   PCLATH
    movwf   Save_PCLATH        ; salva pagina ROM corrente
    movfw   FSR
    movwf   Save_FSR           ; salva puntatore indiretto

    clrf    PCLATH             ; forza pagina 0 per i gestori
                               ; (i gestori sono tutti in pagina 0)

    Nota tecnica: l’uso di SWAPF per salvare STATUS non è casuale. SWAPF non modifica alcun flag di STATUS, a differenza di MOVWF. Questo è cruciale perché l’interrupt può accadere in qualsiasi momento e dobbiamo preservare il contesto esatto.

    Decoder a cascata (fonti interrupt)

    ;----------------------------------------------------------------------
; Le fonti vengono testate in ordine di priorità implicita:
; 1. PORTB change (TLC, SYNC, TACHO, NTC) - più veloci
; 2. Timer2 overflow (timeout ricezione TLC)
; 3. Timer1 overflow (timeout lettura NTC)
; 4. Timer0 overflow (step OS, innesco triac)
;----------------------------------------------------------------------

; --- Fonte 1: PORTB On-Change ---
    btfss   INTCON,RBIF        ; interrupt on change?
    goto    Test_Ovf_Timer     ; no: passa a Timer2

    bcf     INTCON,RBIF        ; resetta flag

Next_Interrupt
    movfw   Last_Pb
    xorwf   PORTB,w            ; quali bit sono cambiati?
    movwf   Save_Port

; Decodifica a catena: ogni gestore può "prendersi" l'interrupt
    SCLR    FL_TLC             ; linea telecomando ?
    goto    Decode_Tlc
End_Test_Tlc

    SCLR    FL_SYNC            ; fronte di sincronismo rete ?
    goto    Decode_Sync

    SCLR    FL_TACHO           ; fronte tachimetrico ?
    goto    Decode_Tacho

    SCLR    FL_NTC             ; fine carica condensatore NTC ?
    goto    Decode_Ntc

End_Test_On_Change

; --- Fonte 2: Timer2 Overflow ---
Test_Ovf_Timer
    btfss   PIR1,TMR2IF        ; overflow timer2 (TLC timeout)?
    goto    Test_Timer1
    ; ... gestione timeout TLC ...

; --- Fonte 3: Timer1 Overflow ---
Test_Timer1
    btfss   PIR1,TMR1IF        ; overflow timer1 (NTC timeout)?
    goto    Test_Timer0
    ; ... gestione timeout NTC ...

; --- Fonte 4: Timer0 Overflow ---
Test_Timer0
    btfss   INTCON,T0IF        ; overflow timer0 (OS step)?
    goto    End_Interrupt
    bcf     INTCON,T0IF

    ; Innesco triac a tempo (zippillo)
    ; ... vedi sezione triac ...

    Innesco triac via interrupt (zippillo a tempo)

    L’interrupt Timer0 viene usato anche per generare l’impulso di gate del triac con temporizzazione precisa:

    ; Dallo stesso interrupt Timer0:
; Diamo lo zippillo al triac richiesto,
; solo se la ventola relativa deve essere accesa.

Test_V_Int_On
    SSET    V_INT              ; test consenso ventola interna
    goto    Test_V_Ext_On
    SSET    V_INT_REQ          ; test richiesta attivazione
    goto    Test_V_Ext_On
    SETB    OK_V_INT           ; consenso attivazione da task!
    CLRB    IO_V_INT           ; azionamento triac (pin basso = ON)

Test_V_Ext_On
    SSET    V_EXT              ; test consenso ventola esterna
    goto    Fire_On_Triac
    SSET    V_EXT_REQ
    goto    Fire_On_Triac
    SETB    OK_V_EXT
    CLRB    IO_V_EXT

Fire_On_Triac
    movlw   7
    movwf   Wait_Rit_Triac     ; attesa 21 us (3 cicli * 7)
Loop_V_xx
    decfsz  Wait_Rit_Triac,same
    goto    Loop_V_xx

    SETB    IO_V_INT           ; disattiva triac (fine impulso)
    SETB    IO_V_EXT

    L’impulso di gate di circa 21 µs è generato interamente via software, senza timer hardware dedicato.

    Schema dei flussi interrupt

    +-----------------------+
| Vettore INT (0x004)   |
+-----------------------+
         |
         v
+-----------------------+
| PUSH: salva contesto  |
| W, STATUS, PCLATH, FSR|
+-----------------------+
         |
         v
+-----------------------+
| PORTB change?         |---SI---> Decode_Tlc
|                       |         Decode_Sync
|                       |         Decode_Tacho
|                       |         Decode_Ntc
+-----------------------+         (catena SCLR/goto)
         |
         NO
         v
+-----------------------+
| Timer2 overflow?      |---SI---> TLC timeout
+-----------------------+         (fine ricezione seriale)
         |
         NO
         v
+-----------------------+
| Timer1 overflow?      |---SI---> NTC timeout
+-----------------------+         (fine carica condensatore)
         |
         NO
         v
+-----------------------+
| Timer0 overflow?      |---SI---> OS step
|                       |         + innesco triac
+-----------------------+         (zippillo a tempo)
         |
         NO
         v
+-----------------------+
| Epilogo: ripristina    |
| contesto, ritorno     |
+-----------------------+

    Lezioni apprese

    • Priorità per latenza: PORTB change in cima (eventi veloci: TLC, sincronismo), timer overflow in fondo
    • Decoder a catena SCLR/goto: ogni gestore "consuma" l’interrupt con SCLR (skip se clear) + goto, realizzando una cascata senza tabelle
    • Zero cicli persi su lettura PORTB: il confronto xorwf PORTB,w col Last_Pb identifica al volo quali pin sono cambiati
    • Epilogi impliciti: non esiste una routine di epilogo centralizzata; ogni gestore termina con retfie dopo aver ripristinato il contesto, o passa al prossimo test
