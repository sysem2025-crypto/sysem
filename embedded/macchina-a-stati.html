Embedded — Progettazione Firmware

    Macchina a Stati per Prodotto

    Diagramma a palle: progettare la logica prodotto in modo che sia leggibile e modificabile dal cliente

    Problema

    Uno split system ha molti modi di funzionamento (Cool, Heat, Dry, Fan, Night, Defrost, Thermo Off, ecc.). Ognuno richiede una combinazione specifica di attuatori: compressore acceso/spento, ventola interna a velocità X, ventola esterna, deflettore in posizione A o B. Il cliente deve poter capire cosa sta facendo la macchina in ogni istante, e il team di sviluppo deve poter aggiungere o modificare stati senza rischiare regressioni. Un tradizionale flow chart non basta: serve un modello che separi chiaramente stato istantaneo delle utenze (output) dalle condizioni di transizione (eventi/tempo).

    Soluzione: Diagramma a Palle

    Ogni stato è rappresentato da un cerchio (“palla”) che contiene la configurazione istantanea di tutte le utenze: compressore, ventole, deflettore, led. Le frecce tra i cerchi rappresentano le transizioni, etichettate con la condizione che le innesca: un evento esterno (es. arriva un comando dal telecomando), un timeout (es. “dopo 2 minuti”), o una condizione sui sensori (es. “T_amb < T_set”).

    Il cliente vede un diagramma a bolle e capisce subito: “Ah, in questo stato il compressore è acceso e la ventola è al minimo”. Lo sviluppatore traduce ogni cerchio in una subroutine Assembly che restituisce il numero dello stato successivo.

    Implementazione: la tabella degli stati (table1.h)

    ; Ogni stato è un'etichetta nella jump table.
; La funzione M_State_Table prende Index_M_S in W
; e salta allo stato corrispondente.
; Ogni stato restituisce in W il prossimo stato.

M_State_Table
    addwf   PCL,same                ; jump table
    nop                             ; parto da 1
    goto    S_1                     ; STOP
    goto    S_2                     ; COOL
    goto    S_3                     ; DRY
    goto    S_4                     ; FAN
    goto    S_5                     ; ...
    goto    S_6                     ; Thermo Off Cool
    goto    S_7                     ; ...
    goto    S_8                     ; Thermo Off Dry
    ...
    goto    S_20                    ; Allarme Hot Battery
    goto    S_23                    ; Allarme NTC
    goto    S_25                    ; Diagnostica ICA
    goto    S_26                    ; Defrost Heat
    goto    S_27                    ; Defrost Heat Night

    Anatomia di uno stato: S_2 (COOL)

    Ogni stato ha tre sezioni ben distinte, che corrispondono esattamente agli elementi del diagramma a palle:

    S_2
; ====================================================================
; 1. CONDIZIONI DI TRANSIZIONE (frecce del diagramma)
;    Codice "on the fly": testa le condizioni di uscita dallo stato.
;    Ogni freccia è un test con goto allo stato destinazione.
; ====================================================================

    ; Freccia: Thermo Off se T_amb raggiunge T_set da 2 minuti
    #ifreg _T_Amb < T_Set
        #ifreg Sat_Timer2 > @SAT_TIMER_1_MIN
            goto    Thermo_Off_Cool_S_6
        #endif
    #endif

    ; Freccia: esci dal COOL se non più in modalità Cool
    SSET    DS_COOL
    goto    Thermo_Off_Cool_S_6

    ; Freccia: protezione antibrinamento dopo 8 minuti
    #ifreg _T_Bat < @T_BAT_NEG_1
        #ifreg Sat_Timer2 > @SAT_TIMER_7_MIN
            goto    Thermo_Off_Cool_S_6
        #endif
    #endif

    ; Freccia: passa a Night se timer notte scaduto
    SSET    DS_NIGHT
    goto    End33
    #ifreg Night_Timer >= @NIGHT_TIMER_1_H
        goto    Cool_Night_S_17
    #endif

    ; Freccia: allarme NTC
    SCLR    ALARM_NTC
    goto    Alarm_Ntc_S_23

; ====================================================================
; 2. STAND CODE (azioni continue nello stato)
; ====================================================================
    clrf    Compr_Timer         ; compressore acceso, reset timer

; ====================================================================
; 3. OUTPUT (stato istantaneo delle utenze - il contenuto del cerchio)
;    Definisce quali attuatori sono ON/OFF in questo stato
; ====================================================================
    SETB    COMPR               ; compressore ON
    ; Ventola esterna: dopo 3 secondi dalla partenza
    #ifreg !Sat_Timer2
        #ifreg Sat_Timer1 >= @3
            SETB    V_EXT
        #endif
    #else
        SETB    V_EXT
    #endif

    SCLR    END_OPEN_FLAP
    SETB    V_INT               ; ventola interna ON

    SCLR    DS_RIC_ARIA
    SETB    RIC_ARIA             ; ricircolo aria ON

    ; Velocità ventola in base alla temperatura
    movlw   V_INT_MIN
    movwf   V_Def
    #ifreg _T_Amb > T_Set + @T_1
        movlw   V_INT_MED
        movwf   V_Def
    #endif
    #ifreg _T_Amb > T_Set + @T_2
        movlw   V_INT_MAX
        movwf   V_Def
    #endif

    SETB    OPEN_FLAP           ; deflettore aperto

    call    Config_Led          ; aggiorna led pannello

    movlw   2                   ; ritorna S_2 (resta in questo stato)
    return

    Legenda frecce:
    ---> transizione a evento (comando utente o allarme)
    - - > transizione a tempo (dopo N secondi/minuti)
    ... > transizione su condizione sensori (temperatura)

    Ogni cerchio dice esattamente cosa accade alle utenze in quello stato. Le frecce dicono quando e perché si passa a un altro cerchio. Il cliente può verificare la logica semplicemente seguendo le frecce. Lo sviluppatore implementa ogni cerchio come subroutine indipendente.

    Il ciclo della macchina a stati

    ; La state machine è una task dello scheduler OS,
; eseguita ogni 250 ms.

State_Machine_Loop_1
    BANK    0
    bcf     INTCON,GIE          ; disabilita interrupt
    EVENT   TASK_M_STATE, MSEC_250
    CLRB    F_MSEC_250

    ; ... lettura configurazione da EEPROM ...

State_Machine_Loop_2
    EVENT   TASK_M_STATE, MSEC_250

    ; ... aggiornamento timer interni ...

    ; Chiamata alla macchina a stati
    movlw   HIGH M_State_Table
    movwf   PCLATH
    movfw   Index_M_S           ; indice stato corrente
    call    M_State_Table       ; esegue stato, torna prossimo stato
    movwf   Index_M_S           ; aggiorna stato corrente

    goto    State_Machine_Loop_1

    Il ciclo è semplice: ogni 250 ms lo scheduler risveglia la task, che chiama lo stato corrente. Lo stato esamina le condizioni, eventualmente aggiorna le utenze, e restituisce il nuovo indice. Se le condizioni non cambiano, lo stato restituisce se stesso (stato invariato).

    Vantaggi del modello a cerchi/frecce

    • Leggibile dal cliente: il diagramma a palle mostra esattamente cosa fa la macchina in ogni stato, senza dettagli implementativi
    • Modificabile dal cliente: “aggiungi una freccia da S_2 a S_6 se T_bat < -1°C per 8 minuti” — il cliente può specificare nuove transizioni in linguaggio naturale
    • Isolamento degli stati: ogni stato è una subroutine indipendente; modificare S_2 non impatta S_10
    • Output autodocumentato: la sezione OUTPUT di ogni stato elenca esplicitamente tutti gli attuatori, rendendo immediato verificare cosa accende/spegne
    • Timer interni per stato: ogni stato ha timer (Sat_Timer1, Sat_Timer2, Roll_Timer) resettati all’ingresso, che permettono transizioni temporizzate senza variabili globali aggiuntive
    • Diagnostica integrata: lo stato corrente viene inviato in seriale per autodiagnosi e debug

    Esempio di modifica richiesta dal cliente

    Il cliente dice: “Nello stato COOL, se la batteria scende sotto -1°C per 8 minuti, voglio che la macchina vada in Thermo Off invece di continuare a raffreddare.”

    Si aggiunge una freccia nel diagramma a palle da S_2 a S_6 con condizione T_bat < -1°C AND timer > 8 min. Nel codice, si aggiungono poche righe nella sezione delle condizioni di S_2:

    ; Freccia: protezione antibrinamento
#ifreg _T_Bat < @T_BAT_NEG_1
    #ifreg Sat_Timer2 > @SAT_TIMER_7_MIN
        goto    Thermo_Off_Cool_S_6
    #endif
#endif

    Nessuna modifica agli altri stati. Il cliente vede la nuova freccia sul diagramma e la verifica sul prodotto reale.

    • Split Elettroplastica I200052 — #M_State.h: 2690 righe, 27 stati (S_1..S_27), di cui 15 operativi + 12 di servizio/diagnostica/allarme

    Entrambi i progetti usano lo stesso modello grafico: cerchi = stati, frecce = transizioni a tempo o evento.
